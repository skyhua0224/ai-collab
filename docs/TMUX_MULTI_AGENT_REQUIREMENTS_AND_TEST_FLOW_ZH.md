# tmux 多 Agent 需求与测试流程（中文）

最后更新：2026-03-04

本文档记录两部分：

1. 用户提出的三轮问题（按轮次归档）。
2. 约定的标准测试流程与验收标准。

## 1）三轮问题记录

## 第一轮：新问题探索

1. 子 Agent 完成状态如何可靠检测？
2. 主控应暂停等用户观察，还是应自动检测完成？
3. 仅靠关键字（如 `TASK_COMPLETE`）为何不可靠：
4. auto-compact 后可能遗忘上下文。
5. 可能误匹配到提示词/命令回显中的同名文本。
6. 轮询策略不稳定会额外消耗主控 token。
7. tmux 状态确认如何标准化并封装为短命令？
8. codex 若在 tmux 外启动，ai-collab 功能还能否使用？
9. 若必须进 tmux，如何“一键回到主控态”且不丢上下文？
10. 已在 tmux 中时，如何绑定 tmux session id 与 ai-collab/controller 会话？
11. 重启后 tmux 丢失时，是否支持 `--resume <id>` 恢复？
12. codex app（非 CLI）或无 tmux 客户端时，协作兜底怎么做？

## 第二轮：架构与可操作性追问

1. JSON 状态是关键词提取还是文件落地？主控如何感知？
2. “暂停等待事件”具体如何等待？超时怎么处理？
3. 是否应由 ai-collab 统一托管等待与超时生命周期？
4. 需要哪些短命令，如何保证 codex/claude/gemini 通用？
5. direct 模式到底有什么价值、边界是什么？
6. codex app 无 tmux 时，多 Agent 协作怎么落地？
7. 能否做到本地可恢复：`ai-collab --resume <id>`？
8. 是否需要 `resume list` 与会话重命名能力？
9. 无 tmux 模式是否有可行方案，需要给出可执行路径。

## 第三轮：执行真实性校验

1. 测试使用了哪个 Agent、哪个模型？
2. 启动时是否发生“早于 shell 就发送输入”？
3. 启动时是否发生“早于 Agent 输入 UI 就发送输入”？
4. 是否出现 `429`？
5. 超过初始超时后，主控是否会自动再次监控？
6. 子 Agent 完成后，ai-collab 监控是否自动采集并推动下一步？
7. 用户修正：超时不是固定 30 秒；第一次可 30 秒，后续必须由主控按状态动态增减。

## 2）标准测试流程（权威基线）

1. 新建 tmux window/session。
2. 启动主控（codex 作为主控角色）在上半屏。
3. 主流程必须用 ai-collab 封装命令（业务流不使用裸 tmux）。
4. 主控分配任务并在下半屏拉起子 Agent（上下 1:1）。
5. 子 Agent 执行一个有明显耗时的任务（测试建议约 1 分钟）。
6. 主控通过 ai-collab watch 监控子 Agent：
7. 第一轮可用 30 秒超时。
8. 后续轮次超时参数由主控动态调整（增/减）。
9. 若未完成，监控结果必须给出结构化原因（running/idle/error 类）。
10. 若卡住，主控可下发继续迭代指令并继续监控。
11. 若模型/服务异常持续，主控需给出降级选项并询问用户决策。
12. 子 Agent 一旦完成，监控应尽快返回完成状态。
13. 主控先询问用户是否关闭已完成 pane。
14. 按用户选择 close/keep 后，再进入下一步流程。

## 3）验收标准

1. 时序正确：先 shell input-ready，再派发命令；先 agent ready，再提交提示词。
2. 无早发输入（不早于 shell/agent 准备完成）。
3. 监控支持超时诊断和结构化原因输出。
4. 监控支持多轮复查且超时由主控动态决策。
5. 完成检测应避免命令回显导致的误判。
6. 子 Agent 完成后，必须先触发“是否关闭 pane”的询问步骤。
7. 关键操作均可通过 ai-collab 命令面完成。

## 4）后续复测记录建议

1. 记录 session id、controller pane id、sub-agent pane id。
2. 保留 handoff/watch/close 全量日志，便于复盘。
3. 若 `--json-output` 带 ANSI 颜色码，解析前先做 ANSI 清洗。
